Interactive Architecture / 交互式架构解读

Teaching AI to Act,
Not Just Think

教 AI 动手做事，而不只是动嘴说话。APEX Environment 是 AI agent 的训练场 — agent 不再只是生成文字，而是学会用 bash 命令解决真实的专业任务：法律分析、投行建模、管理咨询，像一个初级分析师一样工作。

Architecture 架构

Five Layers, One Loop

五层架构，一个循环。点击每一层查看详情，数据从上往下流动。

🎮
Client (ApexEnv) · 客户端
client.py · EnvClient wrapper · Pydantic models

这是 RL trainer 敲的那扇门。Client 把 bash 命令打包成 BashAction 对象，通过 HTTP 发给服务器，再把返回值解包成类型化的 Python 对象（ApexObservation、ApexState）。

它只实现三个方法：_step_payload() 序列化 action，_parse_result() 反序列化 observation，_parse_state() 读取环境状态。

类比：就像电视遥控器。你按按钮（发命令），遥控器把你的意图翻译成红外信号给电视。你永远不用直接碰电视内部。
🌐
API Server · 接口服务
server/app.py · FastAPI · uvicorn

一行代码搞定整个 HTTP API。OpenEnv 的 create_app() 工厂函数接收你的 Environment 类、Action 类型和 Observation 类型，自动生成 /reset、/step、/state、/close 四个端点。

整个文件只有 6 行有效代码。这就是站在 OpenEnv 约定上的力量 — 不写样板代码。

类比：像餐厅的前台系统。你不用从零搭建预约系统、菜单展示和收银台 — 你接入现成的 POS 平台，它自动生成所有界面。你只需要提供菜单（你的环境逻辑）。
🧠
Environment Core (ApexEnvironment) · 核心
server/apex_environment.py · 155 lines · the brain

这是心脏。它管理每个训练 episode 的完整生命周期：

reset() — 抽一个专业任务（比如"分析这笔并购交易"），创建一个全新的临时目录作为 agent 的工作区，把任务说明交给 agent。

step() — 接收一条 bash 命令，在工作区里执行，检查 episode 是否该结束（到了 20 步上限？agent 说了"done"？），如果是，计算奖励。

close() — 清理临时工作区，不留痕迹。

类比：像监考老师。发卷子（reset），看你做题并在每一步检查（step），时间到了收卷（close）。每个考生有自己的桌子（工作区隔离）。
💻
Bash Executor · 命令执行器
server/bash_executor.py · subprocess · 30s timeout

Python subprocess.run() 的轻量包装。接收 bash 命令字符串，在工作目录里执行，捕获 stdout/stderr，强制 30 秒超时。

安全护栏：输出截断到 10KB（stdout）和 5KB（stderr），防止命令输出刷爆内存。超时返回 exit code 124（和 Unix 的 timeout 命令一样）。

类比：像秘书。你写张便条给她，她去终端上敲进去，再把屏幕上显示的东西带回来 — 但如果 30 秒还没跑完，她会直接报告"超时了"。
🎯
Reward Pipeline · 奖励管线
server/reward.py · keyword matching · [0, 1] score

每个 episode 结束时，这个模块给 agent 的工作打分。当前版本（v1）用两个信号：

文件存在性（30%）：Agent 到底有没有创建输出文件？工作区里只要有一个可读文件（.txt, .md, .csv 等），就拿 0.3 分。奖励"做了点什么"而不是"什么都没做"。

关键词覆盖率（70%）：每个任务有评分标准（rubric）。奖励函数从标准里提取重要术语（大写短语、引号里的词），然后检查 agent 的输出文件里出现了多少。匹配越多，分越高。

类比：像老师用 checklist 批改作文。先看："学生到底写了没？"（30%）。再看："他有没有提到评分标准里的关键概念 — 比如'Section 1983'、'IRR 计算'、'市场规模'？"（70%）。不算精读，但足以区分白卷和还行的答案。

Design Decisions 设计决策

Why This Way?

为什么这样设计？每个架构选择背后的思考。

1
Bash, Not MCP · 用 Bash，不用 MCP
为什么选纯 bash 命令而不是结构化工具协议？

MCP（Model Context Protocol）是一个正式协议：你把工具定义成 JSON schema，agent 通过名称和类型化参数来调用。很干净。但它多了很多层：JSON-RPC 传输、工具 schema 定义、参数序列化/验证。

Bash 是通用的。所有 LLM 都已经会写 grep、cat、curl。不需要 schema。不会出"tool not found"错误。Agent 直接输入它在终端里会输入的东西。

Bash (Chosen)

零样本就能用。所有模型都懂 bash。不需要设计 schema。法律、金融、咨询任务用同一个 action space。

Selected

MCP

干净的类型化 API。但每个功能都要写 tool schema。JSON-RPC 开销。模型需要针对特定工具名做训练。

Rejected

Harbor（SkyRL 的训练框架）已经证明这个方案在大规模 RL 训练中可行。

2
Reward: Start Simple, Upgrade Later · 先简单，后升级
三个级别的奖励质量，逐步部署

奖励函数是所有 RL 系统中最重要的部分 — 它定义了什么叫"好"。但一个完美的奖励函数如果要花一周才能写好，还不如一个今天就能跑的"还行"的版本有价值。

v1（当前）：关键词匹配。速度快（零 API 调用），粗糙但能区分好坏。空工作区得 0.0；提到了对的术语的报告得 0.7+。

v2（计划中）：LLM-as-judge。把每条评分标准 + agent 输出发给快速 LLM（Gemini Flash）做逐项评分。复用 Mercor 在 apex-evals 里经过实战检验的评分逻辑。

为什么这个顺序？你得先知道训练管线本身能不能跑通，再去投入奖励质量。如果 RL loop 有 bug，再完美的奖励函数也白搭。先上 v1，验证管线，再升级信号。
3
One Workspace Per Episode · 每个 Episode 一个工作区
通过临时目录隔离，而不是容器

每个 episode 获得自己的 /tmp/apex_{task_id}_{uuid}/ 目录。Agent 只能在这个沙箱里读写。Episode 结束后，目录被删除。

这比每个 episode 开一个 Docker 容器（Harbor 的做法）更轻量，但仍然提供隔离。对于 RL 训练这种需要跑成千上万个 episode 的场景，每个 episode 都起容器的开销太大了。

SUPPORTS_CONCURRENT_SESSIONS = True 意味着多个 episode 可以并行运行 — 每个都有自己的临时目录，互不共享状态。这对批量 RL 训练至关重要，因为你需要同时跑多个 rollout。

4
20-Step Budget + "done" Signal · 20 步预算 + 完成信号
教效率，不只是教正确

Episode 通过两种方式结束：agent 说 "done"，或者到了 20 步上限。这是刻意设计的。

步数限制防止无限循环（一个不停跑 ls 的 agent）。但"done"信号奖励效率 — 一个 5 步就解决任务并说"done"的 agent 比浪费了 20 步的 agent 更好。两个都能拿到奖励，但高效的 agent 省下的算力可以用来跑更多训练 episode。

类比：像一场可以提前交卷的考试。60 分钟到了监考老师会收卷，但如果你 30 分钟就写完交了，你既展示了能力，又给下一个考生让出了教室。

Glossary 术语表

Jargon Buster

这页上的每个技术术语，用人话解释。Every term explained simply.

大局观：RL Trainer 在环境里跑 episode。每个 episode 里，agent 执行动作（bash 命令）并收到观测（终端输出）。最后，奖励函数给 agent 的表现打分。Trainer 用这个分数更新 agent 的大脑（模型权重），让它在下一个 episode 里更聪明。重复几千次。

Reinforcement Learning 强化学习
Environment 环境
AI agent 生活的"世界"。定义了什么动作是可能的、agent 能看到什么、以及怎么评分。
像飞行员的飞行模拟器。不是真的，但表现得像真的 — 飞行员通过在模拟器里练习学会真技术。
Episode 回合
对一个任务的一次完整尝试，从头到尾。从 reset() 开始，到 done=True 结束。
像棋盘游戏的一局。摆好棋盘，玩到有人赢（或时间到），然后重置下一局。
Reward 奖励
一个 0 到 1 的数字，告诉 agent 做得多好。越高越好。只在 episode 结束时计算。
像考试成绩。你不是每做一题就被评分 — 而是交卷后拿到一个总分。
GRPO (Group Relative Policy Optimization)
一个训练算法。让多个 agent 尝试同一个任务，然后强化做得更好的那些。
像拼写比赛。10 个小孩拼同一个单词。拼对的被强化（"继续这样做！"），拼错的被提醒下次换个方式。
APEX Environment
BashAction
Agent 的"棋步" — 一条 bash 命令，比如 ls、cat file.txt 或 echo "analysis" > report.md。
就是在终端里敲一行命令。字面意思。
ApexObservation 观测
执行 bash 命令后返回的东西：终端输出（stdout）、错误信息（stderr）、退出码（0 = 成功）。
就是你在终端里按回车后屏幕上显示的内容。电脑回复你的话。
Workspace 工作区
为每个 episode 新建的临时文件夹（/tmp/apex_...）。Agent 在这里读写文件。Episode 结束后被删除。
像考试时分配给你的干净桌子。一切从空白开始，你在上面工作，考完后清洁工清空。
Rubric 评分标准
附在每个任务上的评分清单。列出 agent 应该涵盖的具体标准，每条带权重（重要程度）。
像辩论赛的评分指南："有没有反驳对方观点？（30%）有没有引用证据？（40%）结论清晰吗？（30%）"
Infrastructure 基础设施
OpenEnv
构建 RL 训练环境的框架。提供基础类和约定，让你不用重新造轮子。
像 Django 之于 web 应用 — 它给你骨架，你填业务逻辑。
FastAPI / create_app()
一个能自动生成 HTTP 端点的 web 框架。create_app() 是 OpenEnv 的魔法函数，把你的环境类变成一个跑起来的 web 服务器。
像一个食品车生成器。你带着菜谱（你的环境），它给你一辆全副武装的餐车，带窗口、菜单板和收银系统。
APEX-v1-extended
HuggingFace 上的数据集，100 个横跨法律、投行和咨询的专业任务。每个任务有说明、评分标准和标准答案。
像一本有 100 道练习题和答案的教材。题目是初级分析师会遇到的真实专业场景。
EnvClient
客户端类，通过 HTTP 和环境服务器通信。在 Python 对象和 JSON 之间来回翻译。
像外交会议上的翻译。Trainer 说 Python，服务器说 HTTP/JSON — 客户端两边都翻。
